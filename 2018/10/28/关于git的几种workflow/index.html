关于git，github，gitlab工作流的学习总结

之前只采用过github工作流以及部分gitlab工作流，对git工作流比较陌生，这次比较系统地学习了下三中工作流，其中git最为复杂，gitlab次之，git最简单

git flow

长期分支master, develop, 短期分支feature, release, hotfix
develop -> feature -> develop
develop -> release -> master & develop
master -> hotfix -> master & develop

关release分支

在git flow中提到创建release分支的时间点，是在develop分支几乎处于一种等待release的状态，这次release所包含的功能都已经合并进develop中，未来新的需要release的功能应该等这次release创建后再合入develop。release分支可以进行测试， 并且bug fix，当release分支可以准备发布的时候， 需要并入master分支， 进行发布，最后release应该merge进develop

关于hotfix分支

当生产环境中出现问题的时候，即master分支，应该由master创建hotfix分支，等到bug fix完成后应该并入master和develop，hotfix的作用就是，其余开发者可以继续基于develop开发，同时可以由一个开发人员对master分支进行fix

gitlab flow

长期分支master, pre-production, production，短期分支feature
master->feature->master->pre-production->production

关于gitlab flow中的hotfix

hotfix可以作为类似feature分支开发（由于下游分支的来源的是master，所以可以基于master进行开发），然后合并进入master进行测试，如果没问题，再合并进入下游分支
但也有可能出现即使通过master上的测试，也不能直接merge或者cherry-pick到下游分支（举个栗子：master已有新的feature合并进来，两个提交修改了同一个地方，合并到下游需要更复杂更严谨的测试），可以发起merge request到下游分支

github flow

长期分支master, 短期分支feature
master->feature->master
